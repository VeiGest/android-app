Requisitos da aplicação móvel
• A aplicação deverá ter, no mínimo, 6 atividades (Activity e/ou Fragment) e um máximo
recomendado de 10 atividades (Activity e/ou Fragment);
o Algumas funcionalidades têm de ser implementadas/apresentadas através de
fragmentos;
• No mínimo, 4 das funcionalidades implementadas têm de aceder ao webservice para
realização das operações CRUD (inserir, consultar, alterar e remover) sobre os dados;
• Será obrigatório a implementação das 4 operações CRUD sobre alguns dos dados acedidos
pelo webservice;
• No mínimo, 2 das atividades implementadas devem permitir introdução de dados pelo
utilizador;
• No mínimo, 2 das atividades implementadas deve ser construída de forma dinâmica, com os
dados obtidos do webservice;
• A aplicação deve conter, pelo menos, um relacionamento master/detail recorrendo aos
serviços desenvolvidos na API;
• É obrigatório o registo de dados em ficheiros locais (shared preferences) e em base de dados
local, em SQL Lite (ex.: cache de informação);
• O acesso aos recursos do webservice não pode bloquear a interface gráfica;
• Será valorizada a utilização de planeamento, regras para coordenação entre elementos do
grupo e implementação que siga as boas práticas (utilização de padrão MVC, comentários,
etc.);
• A aplicação tem de conter um menu de configuração que possibilite a inserção dos dados do
servidor onde está alojada a API.
Adenda
Os projetos entregues para avaliação por época de Exame Normal, devem implementar as seguintes
funcionalidades, para além das definidas no ponto anterior (Requisitos da aplicação móvel):
• Upload de imagens da galeria e da câmara do dispositivo móvel para o servidor web (15%);
• Criação de um ficheiro pdf com dados dinâmicos, que deverá ser enviado por email (5%).

Componentes do Relatório
O relatório do projeto deverá conter os seguintes elementos:
• Introdução do documento
o Contextualização da aplicação;
o Requisitos finais implementados;
o Motivação para o desenvolvimento do projeto;
• Descrição do desenvolvimento
o Metodologia, procedimentos, regras, etc. (ex.: como foi feita a divisão de tarefas);
o Tecnologias usadas e/ou pesquisadas;
o Solução desenvolvida
▪ Descrição dos problemas/dificuldades e soluções;
▪ Justificação das soluções implementadas (ex.: estrutura do código, divisão em
classes, etc., escolha do tipo de menus, etc.)
▪ Identificação das funcionalidades propostas, mas não implementadas;
• Conclusões
o Descrição dos resultados obtidos, problemas por resolver e previsão para o futuro da
aplicação (mesmo que teórico);
A estrutura do relatório deve seguir os pontos definidos acima e representar o trabalho que foi
desenvolvido para chegar ao produto final.
